# 半自动 Freelancing Workflow 结构化总稿 v1

这份总稿的目标不是聊天记录整理，而是作为之后制作 skill 的正式底稿。

## 1. 目标定义

### 核心目标
建立一个**小型、保守、可控、半自动**的 freelancing 工作流，用于：

- 主动发现合适的小单
- 快速筛除模糊、高风险、不适配项目
- 在你确认后进行简洁专业沟通
- 对范围明确的小任务进行执行与交付
- 在项目结束后完成复盘与经验沉淀
- 最终把整套机制打包成可复用 skill

## 2. 系统总原则

### 总原则
**这个 workflow 的目标不是多接单，而是只接清楚、可控、能稳定交付的小单。**

### 优先级排序
1. 准确率优先于数量
2. 稳妥优先于冲动
3. 拒绝不合适的单，和接到合适的单一样重要
4. 不因为想接单而放松标准
5. 宁可错过，也不乱接

## 3. 主流程

### Workflow
**筛选 -> 沟通 -> 确认价格 -> 开工 -> 持续交流与交付 ->（如有问题，售后）-> 复盘 -> 完成目标 -> 总结经验**

## 4. 各阶段定义

### 4.1 筛选
#### 目标
过滤不清晰、不安全、不适配的项目，只留下小型、明确、可快速落地的候选。

#### 核心动作
- 主动找单
- 阅读项目描述
- 阅读客户文档/附件
- 按白名单/黑名单过滤
- 识别客户信号
- 按评分规则打分
- 给出：
  - 推荐
  - 先澄清
  - 拒绝

#### 通过条件
- 项目是小型任务
- 需求明确或基本明确
- 没有红线风险
- 我有足够胜任把握

### 4.2 沟通
#### 目标
通过简洁专业的沟通，确认真实需求、边界和健康度。

#### 核心动作
- 发首次沟通消息
- 提出必要澄清问题
- 验证范围是否稳定
- 识别客户是否存在风险信号

#### 通过条件
- 关键问题已澄清
- 范围足够清楚
- 客户沟通状态健康
- 没有新增重大风险

### 4.3 确认价格
#### 目标
在边界明确后，确认合理的价格和时间预期。

#### 核心动作
- 确认交付物
- 确认任务范围
- 判断报价方式
- 判断时间预估
- 控制承诺强度

#### 通过条件
- 交付边界明确
- 报价建立在清晰信息上
- 时间预估不过度乐观
- 不存在重大未澄清项

### 4.4 开工
#### 目标
在执行前检查通过后，以最小可交付路径开始任务。

#### 核心动作
- 做执行前检查
- 确认没有范围漂移
- 限定当前任务只解决已确认范围
- 按最小闭环执行

#### 通过条件
- 执行前检查全部通过
- 你已确认继续
- 当前任务仍属于小型可控范围

### 4.5 持续交流与交付
#### 目标
在执行中控制风险、同步必要进展，并完成最终交付。

#### 核心动作
- 适度同步进展
- 发现偏差及时说明
- 防止范围膨胀
- 准备交付物
- 做交付前检查
- 你确认后发送交付

#### 通过条件
- 没有失控扩展
- 交付符合约定
- 交付前检查通过
- 你同意发出

### 4.6 售后（如有）
#### 目标
在原范围内处理合理问题，不让售后演变为无限追加工作。

#### 核心动作
- 判断问题是否属于原范围
- 区分缺陷修正与新增需求
- 做小范围补充或明确拒绝扩展

#### 通过条件
- 问题属于原范围内合理修正
- 工作量可控
- 没有改变任务性质

### 4.7 复盘
#### 目标
回看本项目中的判断、执行和风险控制是否得当。

#### 核心动作
- 记录为什么接/拒
- 记录执行中是否出现偏差
- 记录客户和任务表现
- 记录错误判断与成功判断

#### 通过条件
- 已形成项目记录
- 至少识别出一条有效结论

### 4.8 完成目标
#### 目标
让项目形成闭环，不留未定义悬挂状态。

#### 可能结果
- 成功交付并关闭
- 澄清后放弃
- 执行中止损后关闭
- 售后完成后关闭

#### 通过条件
- 项目状态明确结束
- 没有未处理悬挂项
- 记录已完成

### 4.9 总结经验
#### 目标
把单次项目中的经验转化为以后可复用的规则。

#### 核心动作
- 提炼 1–3 条经验
- 判断是否需要更新：
  - 白名单/黑名单
  - 评分规则
  - 沟通模板
  - 报价规则
  - 检查清单
  - 售后边界
- 判断是否写入 skill / references

#### 通过条件
- 至少产出 1 条可复用经验
- 明确经验落点
- 必要时给出规则修正建议

## 5. 行动纪律

1. 先读需求，再表态
2. 先读文档，再判断
3. 先判断能不能做，再想怎么做
4. 不因为想接单而放松标准
5. 不因为报价诱人就忽略风险
6. 不接自己解释不清的项目
7. 不替客户承担未说清的部分
8. 不在模糊状态下承诺交付时间
9. 每次只推进清楚的项目
10. 宁可错过，也不乱接

## 6. 项目类型策略

### 6.1 白名单
#### 小型代码任务
- 小脚本
- 小范围 bug fix
- 数据处理脚本
- 自动化脚本
- 配置修正
- README/文档补全
- 小型网页修改

#### 翻译 / 本地化 / 文本优化
- 中英互译
- 技术文档翻译
- 网站/App 文案本地化
- 邮件润色
- 简介优化
- 说明类文档整理

#### 文档与流程类
- SOP 编写
- 操作说明
- 流程梳理
- FAQ 草稿
- 模板整理

#### 信息整理 / 小型研究类
- 资料汇总
- 链接内容整理
- 竞品信息整理
- 基础研究摘要

### 6.2 灰名单
- 少量调试类代码任务
- 轻度创作文本任务
- 涉及少量外部工具但无高风险权限的任务

灰名单项目必须比普通项目更严格澄清。

### 6.3 黑名单
#### 代码类
- 架构级重构
- 大型系统开发
- 深层疑难 bug
- 生产高风险部署
- 破解、绕过、作弊、刷量
- 长期维护但需求不清

#### 文本类
- 学术代写
- 法律/医疗正式专业文本
- 证件造假
- 虚假宣传
- 需要承担真实性责任的文本

#### 通用
- 模糊单
- 白嫖试工单
- 平台外交易诱导
- 高风险账号操作
- 明显违规任务

## 7. 客户信号识别

### 7.1 绿灯信号
- 描述清晰
- 有文档/附件
- 回答问题直接
- 能说明完成标准
- 边界明确
- 沟通正常
- 预算合理

### 7.2 黄灯信号
- 大体清楚，但信息不完整
- 文档不足
- 有 scope creep 倾向
- 预算偏紧
- 时间偏赶

### 7.3 红灯信号
- 需求模糊且不愿补充
- 不愿提供材料
- 让你先做很多再说
- 明显压价
- 一直变需求
- 诱导平台外沟通/付款
- 甩锅味重
- 灰色要求

## 8. 项目评分规则

每个项目从 6 个维度打分，每项 0–2 分，总分 12 分。

1. 需求明确度
2. 交付物清晰度
3. 任务规模
4. 胜任把握
5. 风险可控性
6. 客户健康度

### 分数解释
- **10–12**：优先推荐
- **7–9**：可保留，但必须先澄清
- **4–6**：不建议
- **0–3**：直接放弃

### 硬红线
即使高分，只要触发以下任一条，也直接否决：
- 困难 bug
- 模糊且文档不足
- 灰产/作弊/侵权
- 高风险权限
- 胜任把握不足

## 9. 澄清规则

### 通用澄清问题
1. 最终希望交付什么？
2. 是否有现成文档、示例、参考文件？
3. 完成标准是什么？
4. 本次边界是什么，不包含什么？
5. 期望交付时间是什么？
6. 是否只需要小范围处理？

### 防 scope creep 问题
1. 这次是否只处理当前描述的部分？
2. 是否不包含额外功能扩展？
3. 是否不包含长期维护？
4. 若发现问题超范围，是否需要重新确认？

## 10. 代码类专用判断规则

### 可接
- 小脚本
- 小范围修复
- 数据处理
- 自动化
- 配置/文档补全

### 直接拒绝
- 困难 bug
- 深层排障
- 高风险部署
- 安全绕过/作弊
- 无法复现的问题
- 需要高权限的任务

### 关键判断问题
1. 语言/框架是什么？
2. 目标是什么？
3. 当前问题是什么？
4. 是否有最小相关代码/文件？
5. 预期输出是什么？
6. 是否只做小范围修改？
7. 是否涉及部署或高权限？

## 11. 翻译 / 文本类专用判断规则

### 可接
- 中英互译
- 技术文档翻译
- 网站/App 文案本地化
- 邮件润色
- 简介优化
- 文本整理

### 直接拒绝
- 法律/医疗专业文本
- 证件正式翻译
- 学术作弊
- 造假内容
- 需要执照担责的文本

### 关键判断问题
1. 原文和目标语言是什么？
2. 用途是什么？
3. 目标受众是谁？
4. 语气风格是什么？
5. 是否有术语表？
6. 只翻译还是要润色/本地化？
7. 最终格式是什么？

## 12. 报价与时间承诺规则

### 报价规则
- 信息不全，不报价
- 模糊任务，不报死价
- 优先按小范围报价
- 不为了接单过度低价

### 时间承诺规则
- 未看清需求，不承诺时间
- 优先给范围型时间预期
- 代码类默认留缓冲
- 复杂度超预期时立即停表态

### 明确禁令
- 没看文档就报价
- 没澄清就报死线
- 为拿单瞎承诺
- 困难 bug 做固定承诺

## 13. 执行前检查清单

### 任务边界
- [ ] 小型任务
- [ ] 目标明确
- [ ] 交付物明确
- [ ] 无隐性大范围工作

### 资料条件
- [ ] 已读需求
- [ ] 已读文档/附件
- [ ] 关键缺失信息已补齐

### 风险判断
- [ ] 非困难 bug
- [ ] 非高风险权限
- [ ] 非灰产/违规/侵权
- [ ] 不需专业资质背书

### 胜任判断
- [ ] 有明确处理路径
- [ ] 不靠碰运气
- [ ] 当前范围在能力内

### 流程确认
- [ ] 你已确认
- [ ] 对客户承诺范围已明确
- [ ] 时间预期不过度

## 14. 交付前检查清单

### 完成度
- [ ] 已完成明确要求部分
- [ ] 无核心遗漏
- [ ] 输出格式符合约定

### 一致性
- [ ] 与客户需求一致
- [ ] 无擅自扩展承诺
- [ ] 无把猜测当结论

### 质量
- [ ] 已做基本自查
- [ ] 无明显错误
- [ ] 逻辑通顺
- [ ] 结果可解释

### 风险
- [ ] 无不该暴露信息
- [ ] 无违规内容
- [ ] 必要声明未遗漏

### 对外交付准备
- [ ] 交付说明清楚
- [ ] 边界限制已说明
- [ ] 你已确认可以发出

## 15. 失败处理机制

### 接错单
- 立即停止继续承诺
- 重新评估范围
- 给你两个选项：缩范围 / 放弃

### 执行中范围超出
- 立刻停下来重判
- 新增内容不默认纳入
- 需要重新确认

### 客户临时变需求
- 单独列出变更点
- 判断是否改变任务性质
- 改变任务性质则重新走接单判断

### 质量不稳
- 不急着交付
- 标记不稳定点
- 不能稳定就不上交

### 客户异常
- 降低优先级
- 暂停投入
- 必要时退出

## 16. 定时找单机制

### 频率
- 每 2–3 小时一次

### 原则
- 宁缺毋滥
- 无合适项目可以汇报“暂无值得推进项目”
- 不为了凑数放松标准

### 每轮数量限制
- 优先推送 1–3 个高质量候选
- 最多不超过 5 个

### 每轮汇报格式
- 时间
- 本轮筛到项目数
- 值得看候选数
- 每个候选的标题、链接、类型、风险点、建议
- 本轮总结

## 17. 项目记录与复盘模板

### 基础项目记录
- 记录时间
- 平台
- 标题
- 链接
- 类型
- 评分
- 客户健康度
- 判断结果
- 主要原因
- 是否触发否决规则
- 你是否同意推进
- 后续结果
- 复盘备注

### 执行项目复盘
- 为什么决定接
- 执行中是否出现：
  - 需求变更
  - 范围膨胀
  - 沟通异常
  - 复杂度超预期
  - 质量不稳
- 最终如何处理
- 哪个判断做对了
- 哪个判断做错了
- 下次应增加什么约束

## 19. 经验总结机制

### 经验模板
- 项目类型
- 这次暴露出的规律
- 以后应怎么处理
- 应修改的流程位置
- 是否写入 skill

### 经验沉淀目标
把单次项目经验转化为：
- 新增限制
- 修改模板
- 更新评分标准
- 更新黑白名单
- 更新 skill/references

## 20. 最终闭环定义

一个项目只有在以下两类结果之一达成时，才算真正闭环：

### 成功完成
- 筛选通过
- 沟通完成
- 价格确认
- 执行完成
- 交付完成
- 售后结束（如有）
- 复盘完成
- 经验已沉淀

### 成功止损
- 在筛选/沟通/执行中发现不适合
- 明确终止
- 项目状态关闭
- 已完成记录与经验总结

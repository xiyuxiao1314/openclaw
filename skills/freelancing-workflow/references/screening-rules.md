# Screening Rules

## Whitelist

Prefer:

- small coding tasks
- small bug fixes without deep uncertainty
- data processing scripts
- automation scripts
- config or README cleanup
- translation and localization
- email or profile polishing
- SOP and process docs
- structured information gathering

## Graylist

Handle more cautiously:

- light debugging that still has reproducible materials
- light creative writing with clear constraints
- work involving external tools but not high-risk accounts or permissions

## Blacklist

Reject:

- large builds or refactors
- difficult bugs or deep debugging
- production-risk deployments
- cheating, abuse, evasion, scraping abuse, or infringement
- legal or medical professional text
- academic cheating
- fake identity or deceptive content
- high-risk account operations
- work with unclear scope

## Client Signals

### Positive
- clear description
- docs or attachments exist
- answers are direct
- scope is bounded
- budget is plausible

### Warning
- missing docs
- weak answers
- time pressure
- budget pressure
- possible scope creep

### Reject Signals
- refuses to clarify
- wants a lot of free work first
- keeps changing the ask
- pushes off-platform payment or communication
- gives strong blame-shifting vibes
- request feels unsafe or deceptive

## Scoring

Score each task 0-2 on:

1. clarity
2. deliverable clarity
3. task size
4. capability confidence
5. risk control
6. client health

### Score Bands
- 10-12: recommend
- 7-9: clarify first
- 4-6: generally avoid
- 0-3: reject

## Red Lines Override Score

Reject regardless of score when:

- ambiguity remains high
- difficult bug is involved
- permissions are risky
- work appears unsafe or deceptive
- confidence is too low
